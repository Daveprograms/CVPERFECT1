# Architecture Decision Records (ADRs)

Short, dated, and *load-bearing*. Each ADR documents a decision and the trade-offs we
explicitly accepted. Future maintainers (including future us) should read these before
proposing changes that contradict them.

---

## ADR-001: Canonical resume content lives in Postgres JSONB

**Status:** Accepted
**Date:** 2026-05
**Context:**
Today the canonical resume is `Resume.content: Text` (raw extracted PDF/DOCX text).
This blocks the template engine, structured editing, ATS scoring, and job-tailored
variants. We need a single, authoritative, structured representation.

**Decision:**
The canonical resume content is a `ResumeDocument` JSON blob stored in
`resumes.document_json` as **Postgres JSONB** (Supabase). The original uploaded file is
kept in object storage for re-extraction; the extracted plain text is kept as
`resumes.source_text` for LLM grounding and search; **but `document_json` is the only
thing the editor and the renderers read or write.**

**Rationale:**
- We already run on Supabase Postgres. JSONB gives us indexable, queryable, transactional
  document storage without introducing a second datastore.
- A separate document store (Mongo, etc.) adds operational surface for no real benefit
  at this stage.
- Event sourcing the editor (append-only operation log + materialized snapshot) was
  considered. Rejected for now: too much infra for a first cut, and we get most of the
  value from per-edit `resume_variants` rows. Can be added later behind the same
  `document_json` interface.

**Consequences:**
- Every read/write path on the resume (editor, AI enhance, AI tailor, render, export)
  goes through `ResumeDocument`. No path may shortcut to `source_text` for editing.
- We must validate the JSON server-side (Pydantic) and client-side (Zod) on every write.
- Migrations need to handle "have `content` but no `document_json` yet" gracefully —
  we backfill on first load (see ADR-006).

## ADR-002: HTML/CSS templates + headless Chromium for PDF export

**Status:** Accepted
**Date:** 2026-05
**Context:**
We need WYSIWYG: the PDF the user downloads must match the live preview pixel-for-pixel.
Three options were considered:
1. HTML/CSS template components + headless Chromium (Playwright) -> PDF.
2. `react-pdf` (React components rendered directly to a custom PDF renderer).
3. LaTeX templates compiled with `xelatex`/`tectonic`.

**Decision:**
Adopt **option 1** as the default rendering pipeline.

**Rationale:**
- Single source of truth: the same React template components render the live preview
  *and* the PDF. There is no "this looks right on screen but breaks in PDF" class of bug.
- Designers and AI-generated themes can iterate in CSS, not in a PDF DSL.
- Chromium gives us real CSS (flex, grid, web fonts, `@page`, `break-inside: avoid`) and
  battle-tested print rendering.
- `react-pdf` rejected: a separate render tree and CSS subset doubles maintenance and
  guarantees preview/export divergence.
- LaTeX rejected: typographically excellent but a *third* rendering path nobody on the
  team wants to maintain, and it locks us out of CSS-based theme authoring. We keep the
  existing `src/*.tex` files as a reference, not as a runtime renderer.

**Consequences:**
- Export becomes a service: a small renderer worker (Node + Playwright) with a strict
  job interface (`renderPdf(variantId, templateId, themeId)`). It is the only Chromium
  consumer in the stack.
- Resource cost: one Chromium process per concurrent export. Sized via the worker pool;
  reused with browser contexts; monitored.
- Web fonts must be self-hosted (no Google Fonts CDN call from the renderer in prod) so
  exports are deterministic and offline-friendly.

## ADR-003: DOCX is rendered from the same `ResumeDocument`, *not* from the PDF

**Status:** Accepted
**Date:** 2026-05
**Context:**
DOCX must be ATS-friendly: real headings, real lists, real text runs — not a PDF-to-DOCX
conversion that produces text boxes.

**Decision:**
DOCX is generated by a dedicated DOCX renderer (Python `python-docx` or Node `docx`
library) that consumes `ResumeDocument + TemplateTheme` and emits a real Word document.
Each template ships **two** renderers (HTML for preview/PDF, DOCX for Word) that share
the same content schema.

**Consequences:**
- Authoring a new template = author HTML/CSS *and* a small DOCX adapter for that template.
  We accept the duplication to guarantee ATS-safe DOCX.
- DOCX templates use a smaller layout vocabulary (single-column, optional two-column via
  table). This is documented in `04_template_engine.md`.

## ADR-004: AI work runs in background jobs; HTTP returns a job id

**Status:** Accepted
**Date:** 2026-05
**Context:**
Resume analyze, enhance, cover-letter, exam, and especially export are too slow and too
flaky for synchronous HTTP. The current "stuck `analyzing`" 409 bug is a direct
consequence of using the request lifecycle as the queue.

**Decision:**
Long-running AI / render work runs in a `background_jobs` queue. The HTTP API for these
features becomes:

```
POST /api/<feature>      -> 202 Accepted, { job_id, status: "queued" }
GET  /api/jobs/{job_id}  -> { status, result?, error? }
```

The frontend polls (or subscribes via SSE/WebSocket later) to the job endpoint.
Synchronous endpoints stay only for trivial operations (CRUD, read-only fetches).

**Consequences:**
- `Resume.processing_status` stops being a lock. It becomes a projection of the latest
  job row for that resume.
- Idempotency keys (`(user_id, feature, content_hash, jd_hash)`) prevent duplicate jobs
  on double-click / StrictMode re-renders.
- Workers can be scaled independently of the API.

## ADR-005: Strict JSON for every AI call, validated by a shared schema layer

**Status:** Accepted
**Date:** 2026-05
**Context:**
Only `analyze_resume_content` enforces JSON MIME and validates output. Everywhere else
relies on heuristic JSON extraction, which fails silently and drifts over time.

**Decision:**
All AI calls go through a shared helper:

```python
ai_runs.call_json(
    kind="resume_analyze",
    schema=ResumeAnalysisAI,         # Pydantic model
    prompt=prompt,
    model="gemini-2.5-flash",
    max_repairs=1,
)
```

The helper:
1. Sends `response_mime_type="application/json"` and `response_schema=...` where supported.
2. Parses + validates against the Pydantic schema.
3. On validation failure, runs **one** repair pass (re-prompt with the validation error).
4. On final failure, raises a typed `AIInvalidResponseError`.
5. Persists an `ai_runs` row with provider, model, prompt hash, latency, tokens, cost,
   status, and result hash.

Free-text outputs (cover letter prose) are still allowed, but they are produced from a
**JSON outline** (also schema-validated) so the structure is auditable even when the
final artifact is prose.

**Consequences:**
- Every AI feature gains observability for free.
- Hallucinations show up as schema validation errors, not as silent UI bugs.
- We can swap providers (OpenAI, Anthropic) by adding an adapter; the call site stays
  the same.

## ADR-006: Backfill `document_json` on first read, not in a one-shot migration

**Status:** Accepted
**Date:** 2026-05
**Context:**
We have existing resumes with `content` but no `document_json`. A one-shot migration
that calls Gemini for every existing row is expensive, slow, and risks rate limits.

**Decision:**
On first access of a legacy resume:
1. If `document_json` is non-null: use it.
2. Else: enqueue a `resume.parse` background job that produces `document_json` from
   `content`, persists it, then completes the original request from the new value.
3. The user sees a "Reformatting your resume…" state for that one access, then the
   resume is forever in the structured world.

**Consequences:**
- No giant batch job at deploy time.
- New uploads always go through the structured pipeline immediately.
- `resumes.document_json IS NULL` is a temporary state with a defined transition.

## ADR-007: One BFF error contract; one frontend data layer

**Status:** Accepted
**Date:** 2026-05
**Context:**
The frontend has at least three response shapes from the BFF and two competing data
layers (manual fetch + React Query). This is the single biggest source of UI bugs.

**Decision:**
- BFF routes return either `{ data: T }` with HTTP 2xx, or `{ error: { code, message,
  details? } }` with the appropriate non-2xx status. **No more 200-with-success-false.**
- All resume / job / analysis flows on the client go through React Query
  hooks (`useResumeQuery`, `useAnalyzeResumeMutation`, etc.) defined in
  `frontend/lib/queries/`. Components do not call `apiService` directly.
- Mutations always invalidate the relevant query keys; in-flight de-dup is handled by
  React Query, not by per-component refs.

**Consequences:**
- The competing manual `useState`/`useEffect` patterns get refactored screen by screen
  in Phase 2 (template engine) and Phase 3 (export).
- The legacy `frontend/components/resume-editor.tsx` axios path gets deleted.

## ADR-008: One ID type (UUID) end-to-end

**Status:** Accepted
**Date:** 2026-05
**Context:**
Mixed UUID and `String(uuid_hex)` columns + `str(current_user.id)` filters cause silent
empty-result bugs and make joins ugly.

**Decision:**
Every primary key and every foreign key in the application schema is `UUID`. All
filters use the typed UUID. All Pydantic schemas use `UUID` for id fields. The frontend
treats them as opaque strings.

**Consequences:**
- One migration converts the offending tables (`jobs`, `interview_*`, `chat_messages`,
  `dream_companies`, `chat`/`watchlist` `user_id` columns) to UUID with FKs.
- Routes drop `str(current_user.id)` casts.
